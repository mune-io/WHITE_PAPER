1. Taxi Advertising Screens

Status: Implemented ‚úÖ

We have deployed a network of digital advertising screens across taxi fleets. The devices serve several key purposes:

  • Advertising platform ‚Äî displaying targeted ads and content based on location, time of day, and passenger profile.
  • Technological backbone ‚Äî tablets act as fully functional computing nodes (Android ARM), capable of future integration into a decentralized network.
  • Local Face Recognition ‚Äî a neural network running entirely on the tablet detects passenger attention and counts ad views. No personal data leaves the device, ensuring compliance with RODO/GDPR.
  • Data privacy ‚Äî all processing is local; only anonymized statistics (ad view counts, engagement metrics) are sent to the server.

This system allows both monetization through ads and preparation for a scalable Web3Taxi ecosystem, with full respect for user privacy.

2. Decentralized Taxi Application

Status: In Progress ‚öôÔ∏è

Core Idea

Instead of a single centralized server, we are building a system where computation (ride-matching, route calculation, driver filtering) is performed on distributed nodes:

  • Advertising tablets in taxis,
  • Stationary mini-servers (Raspberry Pi),
  • Smartphones of drivers and passengers.

Workflow Overview

  1. Passenger places a ride request in the app.
  2. A lightweight Coordinator Server broadcasts the task to the P2P network.
  3. Nearby nodes calculate optimal routes and available drivers.
  4. Results are sent directly to the passenger; the Coordinator only logs key events.

Technical Stack

  • libp2p + WebRTC ‚Äî P2P device communication
  • Redis / Gossip protocol ‚Äî distributed state synchronization
  • DID (Decentralized IDs) ‚Äî identity management
  • Python / Go / React Native ‚Äî node and coordinator implementation

Efficiency Example (Illustrative)

Traditional centralized servers for 10,000 rides/day may require 16 vCPU / 32 GB RAM. Our network of 200 tablets, donating 10‚Äì15% CPU each, can provide ~40 vCPU without additional server costs. This leads to substantial infrastructure cost reduction (~95%).

Note: Estimates are illustrative and not investment advice.

Additional Notes

  • Nodes update location every 2‚Äì3 minutes to keep ride-matching accurate.
  • Passengers and drivers communicate P2P after ride confirmation, minimizing server load.
  • Coordinator only handles essential events (ride acceptance, cancellation, arrival, completion).

3. Offline Network ‚Äî Meshtastic (LoRa)

Status: Planned üîÑ

What is Meshtastic?

Meshtastic implements a mesh network using LoRa technology (not LoRaWAN). Devices relay messages to each other, forming a chain that extends the network coverage.

Key Features

  • Every node retransmits messages, extending coverage.
  • Typical range per node: 2‚Äì5 km in cities, 10‚Äì15 km in open areas.
  • Fully functional city-wide coverage possible with hundreds of nodes, without internet.

Advantages

  • Operates in emergency conditions.
  • No mobile data costs.
  • Low energy consumption for LoRa devices.

4. Internal Payment System

Status: Planned üîÑ

The plan for an internal tokenized payment layer includes:

  • Settlements between drivers, passengers, and advertisers.
  • Advertisement payments via tokens (e.g., MUN).
  • Smart contracts for automated payouts and accounting.
  • Reward programs for resource sharing (computing power, message relays).

Preferred blockchain solutions: Polygon or Base, for low fees and dApp compatibility.

Conclusion

This project aims to create a new type of urban infrastructure, where advertising screens also serve as distributed computing nodes.

  • Network resilience independent of mobile operators
  • Reduced infrastructure costs
  • Fair revenue distribution among participants
